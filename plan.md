 —— 完全自研 CDC 物理拉取引擎

  如果您出于特定业务场景，必须直接与 Observer 进行物理层通信以拉取原始日志，我们将  libobcdc  内部的核心模块进行解耦，为您设计了一套自研 CDC 代替项目的架构蓝图：

    +----------------------------------------------------------------------------+
    |                          您的自研 CDC 客户端                               |
    |                                                                            |
    |   +------------------+     +------------------+     +------------------+   |
    |   | 1. Easy RPC 网络 | --> | 2. 物理CLOG拉取  | --> | 3. GTS事务排序器 |   |
    |   |    客户端通信层   |     |    (LSFetcher)   |     | (Transaction)    |   |
    |   +------------------+     +------------------+     +------------------+   |
    |                                                              |             |
    |                                                              v             |
    |   +------------------+     +------------------+     +------------------+   |
    |   |  输出逻辑事务流  | <-- | 5. 逻辑事务重组  | <-- | 4. Schema 缓存与 |   |
    |   |  (Consistent Tx) |     |  (Reassembler)   |     |    行映像对齐器  |   |
    |   +------------------+     +------------------+     +------------------+   |
    +----------------------------------------------------------------------------+

  要实现这套项目代替  obcdc ，您需要在客户端自主实现以下 5 个硬核底层模块：

  #### 1. 自研 Easy RPC 网络客户端通信层（暂时使用 ob 源码中的代码）

  • 痛点：OceanBase 不使用常规的 gRPC，而是基于自研的  easy  网络库和自定义二进制包协议。
  • 解决：您必须实现其二进制包头解析（包含 Magic Number、Packet Len、Request ID、RPC Type 等），并模拟  async_stream_fetch_log  发送网络帧。

  #### 2. 物理 CLOG 数据拉取与解析 (LSFetcher)

  • 痛点：Observer 发回的是物理-逻辑混合日志（物理的  GroupLogEntry ）。
  • 解决：您需要根据 OceanBase 的日志二进制格式，自行手写解析器（Parser），提取出每个日志项的 LSN、事务 ID（TxID）以及行的二进制前映像/后映像（Before/After Image）。

  #### 3. 全局一致性排序器 (GTS 排序)

  • 痛点：OceanBase 是分布式数据库，日志流（LS）是多分区并发拉取的。直接拿到的日志存在乱序和二阶段提交（Prepare/Commit）断裂。
  • 解决：您必须在内存中设计一个滑动窗口，通过全局事务服务（GTS）提供的提交版本号（Commit Version），对拉取自不同 Observer 节点的日志进行归并排序，还原真实的事务提交顺序。

  #### 4. 动态 Schema 缓存与行映像对齐器

  • 痛点：CLOG 物理日志非常精简，它是不包含列名、甚至不包含数据类型的，只包含物理行偏置值（例如： Col0: 0x0123, Col1: 0x4567 ）。
  • 解决：您的客户端必须通过普通的 SQL 接口，实时查询 OceanBase 的  __all_table 、 __all_column  等系统字典表，在内存中维护一套最新的 Schema 缓存。当解析物理日志时，将物理偏置与 Schema
  进行“动态列对齐”，翻译出真实的表名、列名和数据。
